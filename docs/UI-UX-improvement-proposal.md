# Ghost Writer UI/UX 改善提案 — 最高の体験のために

現状の Ghost Writer は X 風のデザインと明確なステップで使いやすいが、「書くことに没頭できる」「迷わない」「気持ちいい」までを目指すための改善案を整理する。

---

## 1. 第一印象とオンボーディング

### 1.1 初回利用時のガイド
- **現状**: 初めて開いたユーザーに、左のライブラリ・リファレンス・ステップの関係が一目で伝わりにくい場合がある。
- **改善**:
  - **空状態の強化**: 「下書き」「スタイル」「記事」が 0 件のとき、それぞれ「最初のスタイルを作る」「記事をドラッグしてここに」など、次のアクションが分かる一文 + アイコンを表示する。
  - **オプションのツアー**: 「はじめての方」リンクや初回のみの軽いオーバーレイで、「① リファレンスを3つセット → ② 分析 → ③ インタビュー → ④ 記事完成」を 10 秒で説明する。
  - **進捗の言語化**: ステップインジケータの横や下に「リファレンスを揃えましょう」「インタビューで素材を引き出します」など、今何をすればいいか一文で表示する。

### 1.2 ランディングの整理
- ヘッダー周りに「Ghost Writer」の説明を 1 行（例: 「リファレンスのスタイルで、インタビューから記事を生成」）置く。
- モード切替（新規作成 / リライト）を、タブではなくカード選択にして、それぞれの説明文を短く入れると「どちらを選べばいいか」が明確になる。

---

## 2. 視覚と没入感

### 2.1 ダークモード
- **現状**: ライトテーマのみ。
- **改善**: `prefers-color-scheme` またはヘッダーのトグルでダークモードを切り替え。`--bg`, `--surface`, `--text`, `--border` などを CSS 変数で切り替え、記事表示エリアは読みやすいコントラストにする。設定を `localStorage` に保存して次回も維持。

### 2.2 記事エリアの「読む体験」
- **現状**: すでにパディング・行間・フォントサイズは整っている。
- **改善**:
  - 記事表示エリアだけフォントを「読みやすい明朝系」や「Noto Serif JP」などにし、X 記事っぽい「長文を読む」感を強める。
  - 最大幅 700px を維持しつつ、スクロール時にヘッダーやステップバーを薄くする（または固定ヘッダーでコンパクト化）と、記事に集中しやすい。
  - コピー・保存・エクスポートは記事の上か下にまとめ、記事本文と操作を視覚的に分離する。

### 2.3 マイクロインタラクション
- **ボタン**: hover で軽い scale(1.02) や色変化。primary はそのまま、secondary/ghost は押した感じが分かるようにする。
- **ライブラリ項目**: ホバーで「詳細」「削除」が出る現在の仕様を維持しつつ、クリックで「選択された」状態（左ボーダーや背景色）をはっきり示す。
- **ステップ**: 現在の active の下線に加え、ステップ遷移時に 0.2s のフェードでコンテンツが切り替わるようにする（すでに fadeIn があるなら、一貫して使う）。
- **コピー完了**: 「コピーしました」トーストに加え、該当ボタンだけ一瞬「✓ コピーしました」に変えると、何が起きたかが明確になる。

## 3. フィードバックとエラー

### 3.1 トーストの強化
- **現状**: 青いトーストでメッセージ表示。
- **改善**:
  - **種別**: 成功（緑）/ エラー（赤）/ 情報（青）で色とアイコンを変える。例: `showToast("保存しました", "success")`。
  - **持続時間**: 成功は 2〜3 秒で消す、エラーは 5 秒または「閉じる」で消す。
  - **位置**: 画面下部中央のままでもよいが、モバイルでは安全エリア（ノッチ・ホームインジケータ）を避ける。

### 3.2 ローディング状態の統一
- **現状**: スピナーや「執筆中...」オーバーレイがある。
- **改善**:
  - インタビュー・記事生成など「時間がかかる操作」は、**スケルトン**や**進行メッセージ**（「質問を考えています...」「記事を書いています...」）を表示し、ストリーミング開始後は吹き出しにテキストが流れるようにする（すでにストリーミング対応済みなら、その体験を維持）。
  - API エラー時はトースト + 該当エリアに「再試行」ボタンを出すと安心感が増す。

### 3.3 破壊的操作の確認と Undo
- **削除**: スタイル・記事・下書き・テンプレートの削除は、確認ダイアログで「本当に削除しますか？」と明示。可能なら「削除」実行後に「元に戻す」トーストを 5 秒間表示する（バックエンドで論理削除 or 一時退避が必要）。
- **上書き保存**: モーダルで記事を編集したあとの「上書き保存」は、短い確認（「この内容で上書きしますか？」）か、保存成功トーストで十分。バージョン履歴があれば「前の版に戻す」も選べるようにする。

## 4. キーボードと操作効率

### 4.1 ショートカットの明示と統一
- **現状**: チャットで Cmd+Enter 送信など一部あり。
- **改善**:
  - **送信**: インタビュー・リライトのチャット入力で Cmd+Enter（Windows は Ctrl+Enter）で送信（すでにある場合はその旨を入力欄の placeholder や下に小さく表示）。
  - **保存**: Cmd+S で「現在の記事を保存」または「上書き保存」（モーダル開いている場合はモーダル内の保存）。
  - **モーダル**: Escape で閉じる、フォーカスはモーダル内にトラップし、閉じたら元のボタンにフォーカスを戻す。
  - **ヘルプ**: `?` または Cmd+/ でショートカット一覧をモーダル表示する。

### 4.2 フォーカスとタブ順
- ステップを切り替えたあと、そのステップの「最初の入力欄」（タイトル・メモ・チャット入力など）にフォーカスを当てる。
- モーダルを開いたら、タイトルまたは最初の操作可能要素にフォーカス。編集ポップオーバーは入力欄にフォーカス。
- ライブラリのリストは Tab で移動し、Enter で選択できるようにする。

## 5. アクセシビリティ

### 5.1 セマンティクスと ARIA
- ステップインジケータに `role="tablist"` / `tab` / `tabpanel` を付与し、キーボードでタブ切り替え可能にする。
- チャットエリアは `role="log"` または `aria-live="polite"` で、新しいメッセージが読み上げられるようにする。
- 記事出力エリアに `aria-label="生成された記事"` を付ける。見出し構造（h1/h2）はそのまま活かす。
- ボタンには `aria-label` または visible なラベルを付ける（アイコンのみの場合は必須）。

### 5.2 コントラストとフォーカス
- テキストと背景のコントラスト比を WCAG AA 以上に。`--text-dim` や `--text-muted` は背景によっては上げる。
- `:focus-visible` で、キーボードフォーカス時にだけ輪郭（outline または box-shadow）を表示し、マウス操作時は目立たせない。

### 5.3 モバイルとタッチ
- ボタン・ライブラリ項目は最小 44px タッチターゲットにする。
- チャット入力は、モバイルでフォーカス時に画面がスクロールされ、送信ボタンがキーボードの上に来るようにする（必要なら `scrollIntoView`）。

## 6. レスポンシブとレイアウト

### 6.1 狭い画面（〜768px）
- ステップは横スクロールまたは「1/4」のような省略表示にして、常に現在地が分かるようにする。
- 左のライブラリは「開く/閉じる」ドロワーにし、記事・チャットエリアを広く使う。
- 記事のコピー・Markdown・テキストなどボタンは縦に並べるか、折り返して押しやすくする。

### 6.2 広い画面（1400px 以上）
- 最大幅 1400px の中央配置はそのままでよい。記事エリアは 700px 前後に制限し、長文の可読性を保つ。
- オプションで「ライブラリ幅を可変」にすると、多くのスタイル・記事を一覧したいユーザーに便利。

---

## 7. コンテンツとコピー

### 7.1 記事プレビューと X との一貫性
- 記事エリアのレイアウト・フォントを、X 記事のプレビューにできるだけ近づける。これにより「X に貼ったときの見え方」を想像しやすくする。
- 文字数表示（140 / 25,000）はそのまま維持し、閾値超過時は色と一言（「通常ポストの範囲内」など）で分かりやすくする。

### 7.2 コピー後の導線
- 「HTMLでコピー」後に「X記事エディタに貼り付けてください」とトーストで案内する（現状のままでも可）。
- 可能であれば「コピーしました — X で記事を書く」のようなリンク（X の記事作成 URL）をトーストに含め、1 クリックで X を開けるようにする（任意）。

---

## 8. 優先して実装すると効果が大きいもの

1. **トーストの種別（成功/エラー）と持続時間** — 実装コストが低く、フィードバックの質が一気に上がる。  
2. **空状態の文言と CTA** — 初見ユーザーの迷いを減らす。  
3. **キーボードショートカットの統一と「?」で一覧表示** —  power user の満足度が上がる。  
4. **ダークモード** — 長時間利用時の負担を減らし、好みに合わせられる。  
5. **フォーカス管理（ステップ遷移・モーダル）** — キーボードとスクリーンリーダー利用者が使いやすくなる。  
6. **削除の確認と（可能なら）Undo** — 誤操作の不安を減らす。  

これらを段階的に入れることで、「最高の体験」に近づけられる。
